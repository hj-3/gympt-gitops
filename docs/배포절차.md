# 배포 절차

## 초기 배포 (Dev 환경)

### 1. Namespace 생성

```bash
kubectl create namespace backend-api
kubectl create namespace agent-service
kubectl create namespace posture-analysis
kubectl create namespace report-service
kubectl create namespace workers
```

### 2. Argo CD 프로젝트 생성

```bash
kubectl apply -f argocd/projects/gympt-apps.yaml
```

### 3. App-of-Apps 배포

```bash
kubectl apply -f argocd/app-of-apps/dev-apps.yaml
```

### 4. 동기화 대기

```bash
argocd app wait dev-apps --timeout 600
```

### 5. 상태 확인

```bash
kubectl get pods -A
argocd app list
```

## 애플리케이션 업데이트

### 방법 1: 이미지 태그 업데이트 (권장)

CI/CD 파이프라인에서 자동으로 실행:

```bash
cd gympt-gitops
git pull

# Image tag 업데이트
yq eval '.spec.source.helm.parameters[] |= select(.name == "image.tag").value = "dev-abc123"' \
  -i argocd/applications/dev/backend-api.yaml

git add argocd/applications/dev/backend-api.yaml
git commit -m "Update backend-api image to dev-abc123"
git push

# Argo CD가 자동으로 감지하고 동기화 (약 3분)
```

### 방법 2: Helm values 변경

```bash
# values-dev.yaml 수정
vim charts/backend-api/values-dev.yaml

# 예: replicas 증가
# replicaCount: 1 → replicaCount: 2

git add charts/backend-api/values-dev.yaml
git commit -m "Scale backend-api to 2 replicas"
git push
```

### 방법 3: CLI로 수동 동기화

```bash
# 즉시 동기화
argocd app sync backend-api-dev

# Hard refresh
argocd app sync backend-api-dev --force

# Prune
argocd app sync backend-api-dev --prune
```

## 롤백

### 방법 1: Git Revert (권장)

```bash
# 이전 커밋으로 되돌리기
git revert HEAD
git push

# Argo CD가 자동으로 이전 상태로 배포
```

### 방법 2: Argo CD History

```bash
# 배포 히스토리 확인
argocd app history backend-api-dev

# 특정 리비전으로 롤백
argocd app rollback backend-api-dev 5
```

### 방법 3: 자동 롤백

Remediation Worker가 다음 상황에서 자동 롤백:
- Pod 반복 재시작
- 높은 에러율
- Health Check 실패

## Prod 배포

### 1. Dev 테스트 완료 확인

```bash
# Dev 환경 상태 확인
argocd app get backend-api-dev
kubectl get pods -n backend-api
```

### 2. Prod values 업데이트

```bash
# Image tag를 검증된 버전으로 업데이트
vim argocd/applications/prod/backend-api.yaml

# image.tag: "1.0.0"

git add argocd/applications/prod/backend-api.yaml
git commit -m "Deploy backend-api v1.0.0 to prod"
git push
```

### 3. Prod App-of-Apps 배포

```bash
kubectl apply -f argocd/app-of-apps/prod-apps.yaml
```

### 4. 배포 모니터링

```bash
# Argo CD UI에서 확인
# 또는 CLI
argocd app get backend-api-prod --watch

# Pod 상태 확인
kubectl get pods -n backend-api -w
```

### 5. 배포 검증

```bash
# Health Check
curl https://api-prod.gympt.com/actuator/health

# Smoke Test
./scripts/smoke-test.sh prod

# 메트릭 확인
# Grafana에서 API Latency, Error Rate 확인
```

## Canary 배포 (향후)

Argo Rollouts를 사용한 점진적 배포:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
        - setWeight: 20
        - pause: {duration: 5m}
        - setWeight: 50
        - pause: {duration: 5m}
        - setWeight: 100
```

## Blue-Green 배포 (향후)

```yaml
spec:
  strategy:
    blueGreen:
      activeService: backend-api
      previewService: backend-api-preview
      autoPromotionEnabled: false
```

## 긴급 롤백 절차

### 상황: Prod 배포 후 심각한 문제 발생

```bash
# 1. 즉시 이전 버전으로 롤백
argocd app rollback backend-api-prod

# 2. Git에서도 revert
cd gympt-gitops
git revert HEAD
git push

# 3. Slack 알림
# "#incidents" 채널에 상황 보고

# 4. 원인 분석
kubectl logs -n backend-api deployment/backend-api --previous
kubectl describe pod -n backend-api <pod-name>

# 5. 포스트모텀 작성
```
